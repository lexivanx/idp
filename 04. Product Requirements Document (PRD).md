## 4. Product Requirements Document (PRD)

### 4.1 Problem Statement

Product engineering teams in mid-to-large organizations face a persistent bottleneck: the gap between writing application code and running that code reliably in production. This gap manifests as:

1. **Cognitive overhead**: Developers must understand Kubernetes, Terraform, networking, DNS, TLS certificates, secret management, monitoring configuration, CI/CD pipelines, and more — knowledge orthogonal to their primary role.
2. **Inconsistency**: Without standardization, each team invents its own deployment process, monitoring setup, and infrastructure patterns. Debugging across teams requires learning N different approaches.
3. **Toil accumulation**: Repetitive, manual, automatable tasks (environment provisioning, certificate rotation, scaling configuration) consume engineering hours that should go toward product development.
4. **Slow feedback loops**: Provisioning a new environment or service takes days due to ticket-based workflows with human bottlenecks.
5. **Compliance drift**: Security policies exist on paper but enforcement is manual, leading to drift that surfaces only during audits.

### 4.2 Proposed Solution

CloudMultiPloy addresses these problems through five core pillars:

1. **Service Catalog & Golden Paths**: Pre-built, opinionated templates for common service archetypes (REST API, event-driven worker, scheduled job, frontend SPA) that include infrastructure, CI/CD, monitoring, and documentation scaffolding.
2. **Self-Service Portal**: A web UI and CLI where developers can provision services, environments, databases, and other infrastructure resources without filing tickets.
3. **Integrated Ticketing & Workflow**: A built-in ticketing system for requests that require human approval (e.g., production database access, elevated permissions, cost threshold exceptions) with SLA tracking.
4. **Cloud Integration Layer**: A unified abstraction over Azure (MVP) with future extensibility to AWS and GCP, enabling multi-cloud operations without vendor lock-in.
5. **Observability & SRE Dashboard**: Centralized monitoring, SLO tracking, error budget burn-rate visualization, and incident management integration.

### 4.3 Scope Boundaries

**In Scope (MVP — Azure)**:
- Service catalog with 4 golden path templates (Go REST API, Java Spring Boot API, React SPA, Go Worker)
- Self-service provisioning for: AKS namespaces, Azure Database for PostgreSQL, Azure Cache for Redis, Azure Blob Storage, Azure Service Bus
- Ticketing system for approval workflows
- CI/CD pipeline generation (Azure DevOps)
- GitOps deployment (ArgoCD)
- Monitoring stack (Prometheus + Grafana on AKS, integrated with Azure Monitor)
- RBAC and Azure AD integration
- Service mesh (Istio) for inter-service communication
- Cost tracking dashboard (Azure Cost Management API)
- CLI tool for developer workflows

**Out of Scope (MVP)**:
- AWS and GCP cloud providers (post-MVP)
- Custom Kubernetes operator marketplace
- Advanced FinOps features (reserved instance recommendations, spot instance management)
- Mobile application
- Multi-tenancy across separate organizations
