## 10. Data Architecture

### 10.1 Database Strategy

| Service | Database | Justification |
|---------|----------|---------------|
| Catalog Service | PostgreSQL (shared instance, dedicated schema) | Relational data with complex queries; JSONB for flexible metadata |
| Provisioner Service | PostgreSQL (shared instance, dedicated schema) | Terraform state tracking, provisioning history |
| Ticket Service | PostgreSQL (shared instance, dedicated schema) | Workflow state, audit trail; accessed via Spring Data JPA/Hibernate with Flyway migrations |
| Identity Service | PostgreSQL (shared instance, dedicated schema) | User/team/role data |
| Audit Service | PostgreSQL (dedicated instance) | Append-only audit log; may need independent scaling |
| Pipeline Service | PostgreSQL (shared instance, dedicated schema) | Pipeline execution records |
| Cost Service | PostgreSQL (shared instance, dedicated schema) | Cost data aggregation; accessed via Django ORM with Django migrations; pandas used for in-memory analytics |
| Notification Service | Redis | Transient notification queue, delivery status |

**Note on shared vs. dedicated instances**: For MVP, a single PostgreSQL Flexible Server instance with per-service schemas provides simplicity. Each service owns its schema and uses separate connection pools. The schema-per-service approach can be migrated to separate instances later if scaling demands it.

### 10.2 Caching Strategy

- **Redis** is used for: API response caching (5-minute TTL for catalog listings), user session storage, rate limiting counters (sliding window), real-time notification delivery (pub/sub channels), provisioning status polling cache.

### 10.4 Migration Strategy

Database migrations use different tools per service technology:

- **Go services** (Catalog, Provisioner, Identity, etc.): `golang-migrate/migrate` with raw SQL files
- **Java services** (Ticket & Workflow): Flyway with versioned SQL migrations (`V001__create_tickets.sql`), executed automatically on Spring Boot startup via `spring.flyway.enabled=true`
- **Python services** (Cost): Django's built-in migration framework (`python manage.py makemigrations`, `python manage.py migrate`), executed as a Kubernetes init container before the main process starts

```
migrations/
├── catalog/                        # golang-migrate (SQL)
│   ├── 001_create_services.up.sql
│   ├── 001_create_services.down.sql
│   ├── 002_create_templates.up.sql
│   ├── 002_create_templates.down.sql
│   └── ...
├── provisioner/                    # golang-migrate (SQL)
├── identity/                       # golang-migrate (SQL)
├── ticket-service/                 # Flyway (in src/main/resources/db/migration/)
│   ├── V001__create_tickets.sql
│   ├── V002__create_workflow_definitions.sql
│   └── V003__create_approvals.sql
└── cost-service/                   # Django migrations (in costs/migrations/)
    ├── 0001_initial.py
    ├── 0002_add_budget_models.py
    └── 0003_add_anomaly_tracking.py
```

For Go services, migrations run as Kubernetes init containers before service startup, with locking to prevent concurrent migration execution. Java (Flyway) and Python (Django) services handle migration locking natively.
