## 6. Non-Functional Requirements

### 6.1 Performance

| Metric | Target |
|--------|--------|
| Portal page load time (P95) | < 2 seconds |
| API response time (P95) | < 500ms |
| Service creation (end-to-end) | < 10 minutes |
| Resource provisioning (single resource) | < 5 minutes |
| Dashboard rendering (Grafana) | < 3 seconds |
| Concurrent portal users | 200+ |

### 6.2 Availability & Reliability

| Component | SLO | Error Budget (30d) |
|-----------|-----|-------------------|
| Portal & API | 99.95% | 21.6 minutes |
| CI/CD pipeline execution | 99.9% | 43.2 minutes |
| Monitoring stack | 99.99% | 4.32 minutes |
| GitOps sync (ArgoCD) | 99.95% | 21.6 minutes |

### 6.3 Scalability

- The platform shall support horizontal scaling of all stateless components.
- The system shall handle 500+ registered services and 50+ teams without performance degradation.
- Database queries shall be optimized with indexing strategies to maintain < 100ms P99 for catalog queries at scale.
- The Kubernetes control plane shall be sized for 200+ namespaces.

### 6.4 Security

- All data in transit shall be encrypted via TLS 1.3.
- All data at rest shall be encrypted using Azure-managed keys (with customer-managed key option).
- Container images shall be scanned for CVEs before deployment; critical/high CVEs block deployment.
- Network policies shall enforce zero-trust within the cluster (deny-all default, explicit allow).
- The platform shall pass SOC 2 Type II relevant controls.
- Secrets shall never appear in logs, environment variables visible via `kubectl describe`, or Git repositories.

### 6.5 Maintainability

- Code coverage target: > 80% for business logic, > 60% overall.
- All APIs shall be documented via OpenAPI 3.0 specifications.
- Architecture Decision Records (ADRs) shall be maintained for every significant technical decision.
- Runbooks shall exist for every alert and common operational procedure.

### 6.6 Portability

- The platform shall abstract cloud-provider-specific logic behind interfaces to enable multi-cloud support.
- The cloud provider adapter pattern shall be used: Azure adapter (MVP), AWS adapter (post-MVP), GCP adapter (post-MVP).
- All infrastructure shall be defined in Terraform with provider-specific modules behind a common interface module.
