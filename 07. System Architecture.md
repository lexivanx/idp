### 7.2 Architectural Principles

1. **Microservices with Domain Boundaries**: Each core service owns its domain, data store, and API contract. Services communicate asynchronously via Azure Service Bus for commands/events and synchronously via REST (HTTP/JSON) for queries. REST is preferred over gRPC for inter-service communication given the polyglot stack (Go, Java, Python), ensuring every service can communicate without language-specific stub generation overhead.

2. **API-First Design**: Every capability is exposed via a well-documented REST API (OpenAPI 3.0). The web portal and CLI are consumers of the same API — no backdoors.

3. **Cloud-Agnostic Core, Cloud-Specific Adapters**: Core business logic has zero cloud-provider imports. Cloud-specific operations are isolated behind Go interfaces with Azure, AWS, and GCP implementations.

4. **Infrastructure as Code, Everything as Code**: Every piece of infrastructure, configuration, policy, dashboard, and alert is defined in code, version-controlled, and deployed via automated pipelines.

5. **Event-Driven for Extensibility**: Domain events (ServiceCreated, ResourceProvisioned, TicketApproved) are published to Azure Service Bus, enabling loose coupling and future extension points.

6. **Defense in Depth**: Security is layered — network policies, service mesh mTLS, RBAC, API authentication, input validation, secret encryption, audit logging.

#### 7.3.2 Provisioner Service (Go)

**Responsibility**: Orchestrates infrastructure provisioning by executing Terraform plans, managing state, and coordinating async workflows.

**Key Design Decisions**:
- Terraform execution is sandboxed in ephemeral containers (Kubernetes Jobs) to prevent state corruption and ensure isolation.
- Provisioning requests are queued via Azure Service Bus to handle bursts and provide reliable delivery.
- A state machine manages provisioning lifecycle: Pending → Planning → Applying → Verifying → Completed/Failed.
- Terraform state is stored in Azure Blob Storage with locking via Azure Table Storage (one state file per service per environment).

**API Endpoints**:
```
POST   /api/v1/provisions                    — Request resource provisioning
GET    /api/v1/provisions/{id}               — Get provisioning status
GET    /api/v1/provisions/{id}/logs          — Stream provisioning logs
POST   /api/v1/provisions/{id}/cancel        — Cancel in-progress provisioning
GET    /api/v1/provisions/{id}/plan          — Get Terraform plan output
POST   /api/v1/provisions/{id}/approve       — Approve plan for apply
DELETE /api/v1/resources/{id}                — Request resource teardown
GET    /api/v1/resources                     — List provisioned resources
```

#### 7.3.3 Ticket & Workflow Service (Java / Spring Boot)

**Responsibility**: Manages tickets, approval workflows, and SLA tracking. Java was chosen for this service because the JVM ecosystem offers significantly more mature workflow and state machine libraries (Spring Statemachine, Temporal Java SDK) than Go, along with robust enterprise patterns for complex approval chains, SLA timers, and compensation logic.

**Key Dependencies**:
- Spring Boot 3.2+ (Java 21, virtual threads enabled)
- Spring Statemachine for workflow execution
- Spring Data JPA (Hibernate) for persistence
- Spring Security for fine-grained authorization on ticket actions
- Spring Azure Service Bus for event-driven messaging
- Flyway for database migrations
- MapStruct for DTO mapping
- Jakarta Bean Validation for request validation
- Springdoc OpenAPI for API documentation

**API Endpoints**:
```
POST   /api/v1/tickets                       — Create a new ticket
GET    /api/v1/tickets                       — List tickets (filterable)
GET    /api/v1/tickets/{id}                  — Get ticket details
PUT    /api/v1/tickets/{id}                  — Update ticket
POST   /api/v1/tickets/{id}/approve          — Approve ticket
POST   /api/v1/tickets/{id}/reject           — Reject ticket
POST   /api/v1/tickets/{id}/comments         — Add comment
GET    /api/v1/tickets/{id}/history          — Get state transition history
GET    /api/v1/tickets/{id}/sla              — Get SLA status
GET    /api/v1/workflows                     — List workflow definitions
```

**Data Model** (PostgreSQL via JPA/Hibernate):
```java
@Entity
@Table(name = "tickets")
public class Ticket {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(nullable = false, length = 256)
    private String title;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 30)
    private TicketType type; // RESOURCE_REQUEST, ACCESS_REQUEST, CHANGE_REQUEST, INCIDENT, SUPPORT

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 30)
    private TicketState state; // OPEN, TRIAGING, APPROVED, REJECTED, PROVISIONING, COMPLETED, FAILED

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private Priority priority; // LOW, MEDIUM, HIGH, CRITICAL

    @Column(name = "team_id", nullable = false)
    private UUID teamId;

    @Column(name = "reporter_id", nullable = false)
    private UUID reporterId;

    @Column(name = "assignee_id")
    private UUID assigneeId;

    @Column(columnDefinition = "jsonb")
    @JdbcTypeCode(SqlTypes.JSON)
    private Map<String, Object> metadata;

    @Column(name = "sla_response_deadline")
    private Instant slaResponseDeadline;

    @Column(name = "sla_resolution_deadline")
    private Instant slaResolutionDeadline;

    @Column(name = "sla_responded_at")
    private Instant slaRespondedAt;

    @OneToMany(mappedBy = "ticket", cascade = CascadeType.ALL, orphanRemoval = true)
    @OrderBy("createdAt ASC")
    private List<TicketComment> comments = new ArrayList<>();

    @OneToMany(mappedBy = "ticket", cascade = CascadeType.ALL, orphanRemoval = true)
    @OrderBy("createdAt ASC")
    private List<TicketStateTransition> stateHistory = new ArrayList<>();

    @OneToMany(mappedBy = "ticket", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<TicketApproval> approvals = new ArrayList<>();

    @CreationTimestamp
    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt;

    @UpdateTimestamp
    @Column(name = "updated_at", nullable = false)
    private Instant updatedAt;
}
```

**Spring Statemachine Configuration**:
```java
@Configuration
@EnableStateMachineFactory
public class TicketWorkflowConfig extends EnumStateMachineConfigurerAdapter<TicketState, TicketEvent> {

    @Override
    public void configure(StateMachineStateConfigurer<TicketState, TicketEvent> states) throws Exception {
        states.withStates()
            .initial(TicketState.OPEN)
            .state(TicketState.TRIAGING, triagingEntryAction(), null)
            .state(TicketState.APPROVED, approvedEntryAction(), null)
            .state(TicketState.PROVISIONING)
            .end(TicketState.COMPLETED)
            .end(TicketState.REJECTED)
            .state(TicketState.FAILED);
    }

    @Override
    public void configure(StateMachineTransitionConfigurer<TicketState, TicketEvent> transitions) throws Exception {
        transitions
            .withExternal().source(TicketState.OPEN).target(TicketState.TRIAGING)
                .event(TicketEvent.SUBMIT).action(assignToPlatformTeamAction())
            .and()
            .withExternal().source(TicketState.TRIAGING).target(TicketState.APPROVED)
                .event(TicketEvent.APPROVE).guard(quorumApprovalGuard())
            .and()
            .withExternal().source(TicketState.TRIAGING).target(TicketState.REJECTED)
                .event(TicketEvent.REJECT)
            .and()
            .withExternal().source(TicketState.APPROVED).target(TicketState.PROVISIONING)
                .event(TicketEvent.START_PROVISIONING).action(triggerProvisioningAction())
            .and()
            .withExternal().source(TicketState.PROVISIONING).target(TicketState.COMPLETED)
                .event(TicketEvent.PROVISIONING_COMPLETED)
            .and()
            .withExternal().source(TicketState.PROVISIONING).target(TicketState.FAILED)
                .event(TicketEvent.PROVISIONING_FAILED)
            .and()
            .withExternal().source(TicketState.FAILED).target(TicketState.TRIAGING)
                .event(TicketEvent.REOPEN);
    }

    @Override
    public void configure(StateMachineConfigurationConfigurer<TicketState, TicketEvent> config) throws Exception {
        config.withConfiguration()
            .autoStartup(false)
            .listener(new TicketStateMachineListener());  // Publishes events to Service Bus
    }
}
```

#### 7.3.7 Cost Service (Python / Django)

**Responsibility**: Collects, aggregates, and analyzes cloud cost data from Azure Cost Management APIs, provides cost attribution per team/service/environment, manages budgets, and detects cost anomalies. Python was chosen for this service because its data processing ecosystem (pandas, NumPy) is unmatched for the aggregation, trend analysis, and anomaly detection workloads this service performs, and Django provides a rapid development framework with a powerful ORM for complex reporting queries.

**Key Dependencies**:

- Django 5.x with Django REST Framework (DRF)
- Celery + Redis (task broker) + django-celery-beat (periodic scheduling)
