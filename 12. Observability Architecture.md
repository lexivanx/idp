## 12. Observability Architecture

### 12.1 Metrics Pipeline

```
Application Pods ──(Prometheus SDK)──→ /metrics endpoint
        │
kube-state-metrics ──────────────────→ Kubernetes object metrics
node-exporter ───────────────────────→ Node hardware metrics
Istio Envoy proxies ─────────────────→ Service mesh metrics
        │
        ▼
Prometheus Server (scrape every 15s)
        │
        ├──→ Alertmanager ──→ PagerDuty / Slack / Email
        │
        ├──→ Grafana (dashboards)
        │
        └──→ Thanos Sidecar ──→ Azure Blob Storage (long-term retention)
```

### 12.2 Logging Pipeline

```
Application Pods (stdout/stderr)
        │
        ▼
Fluent Bit (DaemonSet on every node)
        │
        ├──→ Azure Log Analytics (structured JSON logs)
        │
        └──→ Grafana Loki (optional, for unified Grafana experience)
```

### 12.3 Distributed Tracing

```
Application Pods (OpenTelemetry SDK)
        │
        ▼
OpenTelemetry Collector (Deployment)
        │
        ├──→ Azure Application Insights
        │
        └──→ Jaeger (self-hosted for detailed trace analysis)
```

### 12.4 SLO Configuration

```yaml
# PrometheusRule for SLO-based alerting
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: catalog-service-slo
  namespace: cloudmultiploy-system
spec:
  groups:
    - name: catalog-service-slo
      rules:
        # Availability SLO: 99.95%
        - record: catalog_service:availability:rate5m
          expr: |
            1 - (
              sum(rate(http_requests_total{service="catalog-service",code=~"5.."}[5m]))
              /
              sum(rate(http_requests_total{service="catalog-service"}[5m]))
            )
        
        # Error budget burn rate (1h window)
        - alert: CatalogServiceHighBurnRate
          expr: |
            (
              1 - catalog_service:availability:rate1h
            ) / (1 - 0.9995) > 14.4
          for: 5m
          labels:
            severity: critical
            team: platform
          annotations:
            summary: "Catalog service error budget burn rate is 14.4x normal"
            runbook: "https://cloudmultiploy.company.com/runbooks/catalog-service-high-error-rate"
        
        # Latency SLO: P99 < 500ms
        - alert: CatalogServiceHighLatency
          expr: |
            histogram_quantile(0.99,
              sum(rate(http_request_duration_seconds_bucket{service="catalog-service"}[5m])) by (le)
            ) > 0.5
          for: 10m
          labels:
            severity: warning
            team: platform
```

### 12.5 Grafana Dashboard Definitions

Dashboards are defined as JSON and managed via Grafana's provisioning system (ConfigMaps mounted into Grafana):

```
monitoring/
├── grafana/
│   ├── dashboards/
│   │   ├── platform-overview.json        # High-level platform health
│   │   ├── service-detail.json           # Per-service RED metrics
│   │   ├── kubernetes-cluster.json       # Cluster resource utilization
│   │   ├── slo-overview.json             # SLO compliance across services
│   │   ├── deployment-tracking.json      # Deployment frequency, lead time
│   │   ├── cost-attribution.json         # Cloud spend per team/service
│   │   ├── ticket-metrics.json           # Ticket SLA compliance
│   │   └── provisioner-health.json       # Terraform execution metrics
│   └── provisioning/
│       ├── dashboards.yaml               # Dashboard provisioning config
│       └── datasources.yaml              # Prometheus/Loki datasource config
```
