## 14. SRE & Scalability Strategy

### 14.1 SRE Principles Applied

1. **SLOs as First-Class Citizens**: Every service has defined SLOs (availability, latency). Error budgets drive deployment velocity decisions — when budget is exhausted, focus shifts to reliability.

2. **Toil Budget**: The platform team allocates no more than 50% of time to toil. The ticketing system tracks toil-related tickets to measure and reduce over time.

3. **Blameless Post-Mortems**: Incident management includes structured post-mortem templates committed to the repository, with action items tracked as GitHub Issues.

4. **Progressive Rollouts**: All production deployments use canary releases via Argo Rollouts with automated metric analysis:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: catalog-service
spec:
  replicas: 5
  strategy:
    canary:
      canaryService: catalog-service-canary
      stableService: catalog-service-stable
      trafficRouting:
        istio:
          virtualServices:
            - name: catalog-service-vsvc
              routes:
                - primary
      steps:
        - setWeight: 10
        - pause: { duration: 5m }
        - analysis:
            templates:
              - templateName: success-rate
            args:
              - name: service-name
                value: catalog-service
        - setWeight: 30
        - pause: { duration: 10m }
        - analysis:
            templates:
              - templateName: success-rate
        - setWeight: 60
        - pause: { duration: 10m }
        - setWeight: 100
```

5. **Chaos Engineering**: Scheduled chaos experiments using Chaos Mesh to validate resilience assumptions:
    - Pod kill (random service instance termination)
    - Network partition (simulate AZ failure)
    - CPU/memory stress (validate HPA scaling)
    - DNS failure (validate retry logic)

### 14.2 Scalability Design

| Component | Scaling Strategy | Mechanism |
|-----------|-----------------|-----------|
| Web Portal | Horizontal | HPA on CPU/memory, CDN for static assets |
| API Services | Horizontal | HPA on custom metrics (request rate), PDB for availability |
| Provisioner | Queue-based | Scale workers based on Service Bus queue depth |
| PostgreSQL | Vertical + Read Replicas | Azure Flexible Server scaling, read replica for reports |
| Redis | Vertical + Clustering | Azure Cache for Redis Premium with clustering |
| Prometheus | Federation + Thanos | Thanos for long-term storage and global view |
| ArgoCD | Shard by cluster | Application controller sharding |

### 14.3 Disaster Recovery

| Tier | RPO | RTO | Strategy |
|------|-----|-----|----------|
| Platform Database | 5 min | 30 min | Azure geo-redundant backup, point-in-time restore |
| Terraform State | 0 (sync) | 15 min | Azure Blob GRS, versioning enabled |
| GitOps Repos | 0 (sync) | 5 min | GitHub (inherently distributed) |
| AKS Cluster | N/A | 2 hours | Re-provision via Terraform + ArgoCD re-sync |
| Monitoring Data | 1 hour | 1 hour | Thanos to Azure Blob, Grafana dashboard-as-code |
