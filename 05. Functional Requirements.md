## 5. Functional Requirements

### 5.1 Service Catalog & Golden Paths

**FR-SC-001: Template Registry**
- The system shall maintain a versioned registry of golden path templates.
- Each template shall define: application scaffolding, Dockerfile, Kubernetes manifests (Helm charts), CI/CD pipeline definition, Terraform modules for required infrastructure, Prometheus alerting rules, Grafana dashboard definitions, and README documentation.
- Templates shall be stored as Git repositories and versioned using semantic versioning.
- Platform engineers shall be able to create, update, deprecate, and archive templates via the admin UI and CLI.

**FR-SC-002: Service Creation Wizard**
- The portal shall provide a step-by-step wizard for creating new services from golden path templates.
- Wizard steps: (1) Select template, (2) Configure service metadata (name, team, description, tier), (3) Select infrastructure components (database, cache, message queue), (4) Configure environment strategy (dev, staging, production), (5) Review and confirm.
- Upon confirmation, the system shall: create a Git repository with scaffolded code, provision infrastructure via Terraform, configure CI/CD pipeline, register service in the service catalog, create initial Grafana dashboards and Prometheus alerts.
- The entire creation process shall complete within 10 minutes for standard templates.

**FR-SC-003: Service Catalog Browser**
- The portal shall display all registered services with: name, owning team, template used, deployment status per environment, health status (SLO compliance), last deployment timestamp, dependencies, and documentation links.
- Users shall be able to filter by team, template, status, and tier.
- Each service detail page shall provide: architecture diagram (auto-generated from dependencies), environment URLs, deployment history, SLO status, cost attribution, runbook links, and on-call schedule.

**FR-SC-004: Dependency Mapping**
- The system shall automatically discover and visualize service dependencies based on: Istio service mesh telemetry, Kubernetes service discovery, explicit declarations in service metadata.
- Dependency graphs shall be rendered interactively with drill-down capability.

### 5.2 Self-Service Infrastructure Provisioning

**FR-SS-001: Resource Provisioning Engine**
- The system shall support self-service provisioning of the following Azure resources: AKS namespaces with RBAC, Azure Database for PostgreSQL (Flexible Server), Azure Cache for Redis, Azure Blob Storage accounts, Azure Service Bus namespaces and queues/topics, Azure Key Vault instances, Azure DNS zones and records.
- Each resource type shall have configurable tiers (small, medium, large) with pre-defined sizing that aligns with organizational cost policies.
- Provisioning shall be idempotent and state-tracked via Terraform state stored in Azure Blob Storage with state locking via Azure Table Storage.

**FR-SS-002: Environment Management**
- The system shall support environment lifecycle management: create, update, clone, teardown.
- Standard environments: development, staging, production.
- Users shall be able to create ephemeral environments for feature branches that auto-delete after a configurable TTL (default: 7 days).
- Environment promotion (dev → staging → production) shall be a single-click operation with mandatory approval gates for production.

**FR-SS-003: Secret Management**
- The system shall integrate with Azure Key Vault for secret storage.
- Secrets shall be injected into Kubernetes pods via the Azure Key Vault Provider for Secrets Store CSI Driver.
- Secret rotation shall be automated for database credentials, API keys, and TLS certificates.
- The system shall maintain an audit trail of all secret access and modifications.

### 5.3 Ticketing & Workflow System

**FR-TK-001: Ticket Types**
- The system shall support the following ticket types: Resource Request (infrastructure provisioning outside self-service tiers), Access Request (elevated permissions, production database access), Change Request (infrastructure modifications requiring approval), Incident Report (linked to monitoring alerts), General Support (platform usage questions, troubleshooting).

**FR-TK-002: Workflow Engine**
- Each ticket type shall have a configurable workflow with states, transitions, and approval rules.
- Approval rules shall support: single approver, multi-approver (all must approve), quorum (N of M must approve), auto-approve (based on conditions like resource cost < threshold).
- Workflows shall be defined as declarative YAML and version-controlled.
- The system shall enforce SLA tracking for each ticket type with configurable response and resolution targets.

**FR-TK-003: Notifications & Integration**
- The system shall send notifications via: email, Slack/Teams webhook, in-app notification center.
- Tickets shall support comments, attachments, and status change history.
- The system shall expose a webhook API for external integrations.

### 5.4 CI/CD Pipeline Management

**FR-CI-001: Pipeline Generation**
- Upon service creation, the system shall generate an Azure DevOps pipeline definition (YAML) tailored to the selected golden path template.
- Pipeline stages shall include: lint, unit test, integration test, SAST (static application security testing), container image build, container image scan, push to Azure Container Registry, deploy to dev (automatic), deploy to staging (automatic with integration test gate), deploy to production (manual approval gate).

**FR-CI-002: GitOps Deployment**
- Production deployments shall follow GitOps principles using ArgoCD.
- The system shall maintain a separate GitOps repository per environment containing Helm value overrides.
- ArgoCD shall be configured for auto-sync in dev/staging and manual-sync in production.
- The system shall support canary deployments and progressive rollouts via Argo Rollouts.

**FR-CI-003: Pipeline Observability**
- The portal shall display pipeline execution history, duration trends, failure rates, and flaky test detection.
- Users shall be able to trigger pipeline re-runs, view logs, and download artifacts from the portal.

### 5.5 Monitoring & Observability

**FR-MO-001: Metrics Collection**
- The system shall deploy Prometheus on AKS for metrics collection.
- All golden path services shall emit standard metrics: request rate, error rate, latency (RED metrics), and saturation metrics.
- Infrastructure metrics shall be collected via kube-state-metrics, node-exporter, and Azure Monitor.

**FR-MO-002: Dashboarding**
- Grafana shall be deployed as the dashboarding solution.
- Golden path templates shall include pre-built Grafana dashboards for: service RED metrics, Kubernetes pod/node resource usage, database connection pool and query performance, cache hit rates, message queue depth and processing latency.
- Users shall be able to create custom dashboards.

**FR-MO-003: Alerting**
- Alerts shall be defined as PrometheusRule CRDs and managed via the golden path templates.
- Alert routing shall be configured via Alertmanager with team-based routing.
- The system shall integrate with PagerDuty/OpsGenie for on-call notification.
- Alert suppression during maintenance windows shall be supported.

**FR-MO-004: SLO Management**
- The system shall allow teams to define SLOs for their services (availability, latency percentiles).
- Error budget burn-rate dashboards shall be auto-generated.
- The system shall emit alerts when error budget burn rate exceeds configurable thresholds (e.g., 2x normal burn rate over 1 hour).

### 5.6 Administration & RBAC

**FR-AD-001: Identity & Access Management**
- The system shall integrate with Azure Active Directory (Entra ID) for authentication via OIDC.
- RBAC roles: Platform Admin (full access), Team Lead (manage team's services, approve tickets), Developer (self-service within team scope), Viewer (read-only access).
- Permissions shall be scoped to teams and services.
- All actions shall be logged in an immutable audit trail.

**FR-AD-002: Team Management**
- Platform admins shall be able to create teams, assign members, and configure team-level policies (resource quotas, allowed templates, cost budgets).
- Teams shall map to Kubernetes namespaces and Azure resource groups for isolation.

**FR-AD-003: Platform Configuration**
- Platform admins shall be able to configure: cloud provider credentials (Azure service principals), default resource tiers and quotas, approval workflow rules, notification channels, SLA targets for ticket types.
